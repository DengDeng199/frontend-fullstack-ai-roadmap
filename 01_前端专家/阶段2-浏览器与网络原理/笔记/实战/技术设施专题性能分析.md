## 一、系统概况

### 1.1 页面特征分析（基于视觉观察）

| 特征项 | 观察结果 | 潜在性能影响 |
|--------|---------|-------------|
| **页面类型** | 数据可视化大屏（Dashboard） | 大量 DOM 节点、频繁重绘 |
| **可视化组件** | 地图（ECharts/高德）、柱状图、仪表盘、统计卡片 | Canvas/WebGL 渲染消耗 GPU |
| **布局复杂度** | 多列布局，大量绝对定位元素 | 布局计算复杂 |
| **动画效果** | 数字滚动、进度条动画、地图交互 | 可能导致布局抖动 |
| **实时数据** | 在线率、使用率等实时指标 | 频繁 setState 触发重渲染 |

### 1.2 典型大屏系统性能风险点

```
┌─────────────────────────────────────────────────────────────┐
│                    大屏系统性能风险热力图                      │
├─────────────────────────────────────────────────────────────┤
│  🔴 高风险                                                   │
│     ├── 地图组件（大量标记点 + 连线）→ 重绘开销大             │
│     ├── 实时数据轮询（WebSocket/轮询）→ 内存泄漏风险          │
│     └── 数字滚动动画（requestAnimationFrame 未节流）          │
│                                                             │
│  🟡 中风险                                                   │
│     ├── 图表组件过多（ECharts 实例未复用）                    │
│     ├── 图片资源未压缩（背景图、图标）                        │
│     └── 未使用虚拟滚动（数据量大时）                          │
│                                                             │
│  🟢 低风险（但需关注）                                        │
│     ├── CSS 动画（transform > left/top）                     │
│     └── 字体文件加载                                         │
└─────────────────────────────────────────────────────────────┘
```

## 二、Chrome DevTools Performance 操作指南

### 2.1 录制前准备

#### 步骤 1：打开无痕模式（排除插件干扰）

```
快捷键：Ctrl + Shift + N
```

#### 步骤 2：打开 DevTools Performance 面板

```
方式1：F12 → Performance 标签
方式2：Ctrl + Shift + P → 输入 "Show Performance"
```

#### 步骤 3：配置录制参数

```
Performance 面板右上角设置（齿轮图标）：
├── CPU: 4x slowdown（模拟低配设备，暴露性能问题）
├── Network: Fast 3G（模拟弱网环境）
└── ✅ 勾选 "Screenshots"（记录渲染过程截图）
```

> **为什么 CPU 降速？** 你的开发机配置通常高于用户设备，4x slowdown 能暴露更多性能问题。

### 2.2 录制流程（关键！）

#### 录制场景 1：页面首次加载

```
1. 按 Ctrl+Shift+E 或点击 "Record" 按钮
2. 立即按 F5 刷新页面
3. 等待页面完全渲染（所有图表加载完毕，数字停止滚动）
4. 点击 "Stop" 停止录制
```

#### 录制场景 2：交互响应（地图交互）

```
1. 点击 "Record"
2. 在地图上执行以下操作：
   - 点击某个地市标记点
   - 缩放地图
   - 切换顶部 Tab（技术设施 → 频谱资源）
3. 等待 3-5 秒
4. 点击 "Stop"
```

#### 录制场景 3：长时间运行（内存泄漏检测）

```
1. 切换到 Memory 面板
2. 点击 "Take heap snapshot"（记录初始内存）
3. 让页面运行 5-10 分钟（或执行多次数据刷新）
4. 再次点击 "Take heap snapshot"（记录最终内存）
5. 对比两次快照，查看内存增长
```

```
│                                                              │

│  ├─┤  ├────┤  ├─┤              每个条代表一个请求             │
│                                                              │
│  Screenshots (截图序列)                                       │
│  [缩略图] [缩略图] [缩略图]     可查看每一帧的渲染状态         │
└──────────────────────────────────────────────────────────────┘
```

**CPU 颜色含义**：

| 颜色 | 含义 | 优化建议 |
|------|------|---------|
| 🟡 黄色 | JavaScript 执行 | 减少复杂计算、使用 Web Worker |
| 🟣 紫色 | 样式计算/重排 | 批量修改样式、使用 transform |
| 🟢 绿色 | 绘制/合成 | 减少重绘区域、使用 will-change |

### 3.2 Main 主线程分析（最关键）

```
Main Thread (主线程)
├── 🔥 Task (长任务，超过 50ms 需关注)
│   ├── Parse HTML
│   ├── Evaluate Script (JS 执行)
│   ├── Recalculate Style (样式计算)
│   ├── Layout (布局/重排)
│   ├── Paint (绘制)
│   └── Composite (合成)
│
├── 🔴 Long Task (长任务警告)
│   └── 超过 50ms 的任务会阻塞主线程，导致卡顿
│
└── 🟡 Forced Reflow (强制同步布局)
    └── 在读取 offsetHeight 后立即修改样式，导致强制重排
```

**关键诊断步骤**：

```
1. 查看是否有红色三角形警告（Long Task）
2. 点击长任务，查看 Call Tree（调用栈）
3. 定位到具体的函数名和源码位置
4. 分析是业务逻辑问题还是框架问题
```

### 3.3 Frames 帧分析

```
Frames (每一帧的渲染情况)
├── 🟢 绿色：正常渲染（16.67ms 内完成，60fps）
├── 🟡 黄色：轻微延迟（16-33ms，30-60fps）
└── 🔴 红色：严重掉帧（>33ms，<30fps）

点击任意帧可查看：
- Duration (耗时)
- Screenshot (该帧画面)
- Layer tree (图层树)
```

### 3.4 Network 网络分析

```
Network (瀑布流)
├── 查看资源加载顺序
├── 识别阻塞渲染的资源（Render-blocking）
└── 检查资源大小和加载时间

重点关注：
- 是否有资源加载超过 1s
- JS/CSS 是否阻塞了首次渲染
- 图片是否未经压缩
```

## 四、针对本系统的专项检测清单

### 4.1 地图组件性能检测

```
检测方法：
1. 录制地图交互操作（点击标记点、缩放）
2. 查看 Main 线程是否有长任务

常见问题：
□ 地图标记点过多（>500 个）→ 使用聚合（Cluster）
□ 连线动画使用 setInterval → 改为 requestAnimationFrame
□ 地图实例未销毁（切换 Tab 时）→ 调用 dispose()

优化方案：
├── 标记点聚合：当地图缩放级别较小时，将邻近点合并显示
├── 懒加载：视口外的标记点延迟渲染
└── Canvas 渲染：大数据量时启用 Canvas 模式（ECharts）
```

### 4.2 图表组件性能检测

```
检测方法：
1. 查看页面加载时 ECharts 初始化耗时
2. 检查数据更新时是否有卡顿

常见问题：
□ 图表实例过多（每个卡片一个 ECharts 实例）
□ setOption 全量更新 → 改为增量更新
□ 动画时长过长 → 减少 animationDuration

优化代码示例：
// ❌ 错误：每次都全量更新
myChart.setOption({
  series: [{ data: newData }]
});

// ✅ 正确：增量更新 + 禁用动画
myChart.setOption({
  series: [{ data: newData }]
}, {
  notMerge: false,
  lazyUpdate: true
});
```

### 4.3 数字滚动动画检测

```
检测方法：
1. 查看是否有持续的 requestAnimationFrame 调用
2. 检查数字停止后是否还在执行动画逻辑

优化方案：
├── 使用 CSS animation 替代 JS 动画
├── 数字变化时使用 transform 而非修改 innerText
└── 组件卸载时取消动画（避免内存泄漏）
```

### 4.4 内存泄漏检测

```
Memory 面板操作：
1. 点击 "Take heap snapshot"
2. 执行可能导致泄漏的操作（如切换 Tab 10 次）
3. 再次点击 "Take heap snapshot"
4. 切换到 "Comparison" 视图
5. 查看 #Delta 列，正值表示内存增长

常见泄漏原因：
□ 事件监听器未移除（addEventListener 未对应 remove）
□ 定时器未清理（setInterval/setTimeout）
□ ECharts 实例未 dispose
□ 闭包引用外部变量导致无法 GC
```

## 五、预期瓶颈与优化建议（基于视觉分析）

### 5.1 高风险优化项

| 问题 | 优化前（预估） | 优化方案 | 优化后目标 |
|------|--------------|---------|-----------|
| **地图标记点过多** | 157 个区县点全部渲染 | 视口内渲染 + 聚合 | 同时渲染 <50 个点 |
| **实时数据轮询** | 每 5s 全量更新所有图表 | WebSocket + 增量更新 | 只更新变化的数据 |
| **数字滚动动画** | requestAnimationFrame 持续执行 | CSS animation + 节流 | 减少 80% JS 执行 |
| **ECharts 实例过多** | ~10 个独立实例 | 实例复用 + 懒加载 | 减少 50% 内存占用 |

### 5.2 中风险优化项

```
优化项清单：

□ 图片资源优化
  ├── 背景图使用 WebP 格式
  ├── 小图标使用 SVG Sprite
  └── 大图使用 lazy loading

□ CSS 优化
  ├── 避免使用 :hover 触发重排
  ├── 使用 transform 替代 top/left 动画
  └── 对固定元素使用 contain: layout

□ JavaScript 优化
  ├── 防抖/节流处理 resize/scroll 事件
  ├── 大数据量使用虚拟滚动
  └── Web Worker 处理复杂计算
```

## 八、附录：常见问题排查

### Q1：Performance 面板卡顿无法录制？

```
解决方案：
1. 减少录制时间（控制在 10s 以内）
2. 关闭 Screenshots 选项
3. 降低 CPU slowdown 倍数
```

### Q2：无法定位到具体源码位置？

```
解决方案：
1. 确保生产环境有 source map
2. 在 DevTools → Settings → Preferences 中启用 "Enable JavaScript source maps"
3. 如果是压缩代码，使用本地开发环境复现
```

### Q3：如何导出性能报告？

```
操作步骤：
1. Performance 面板右上角 → 下载图标
2. 保存为 .json 文件
3. 可在其他 Chrome 中导入查看（点击上传图标）
```

### Q4：地图组件在 Performance 中显示为 Long Task？

```
优化方向：
1. 使用 Web Worker 处理地图数据
2. 分批渲染标记点（requestIdleCallback）
3. 使用 Canvas 渲染替代 DOM 渲染
```
