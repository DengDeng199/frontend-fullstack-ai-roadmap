# 浏览器渲染流水线全流程（HTML→DOM→渲染）核心知识点\+实操案例

## 一、整体渲染链路总览（必考主线）

**完整流水线顺序（不可逆）**：HTML → DOM 树 → CSSOM → Render Tree → Layout\(回流\) → Paint\(绘制\) → Composite\(合成\)

核心结论：浏览器从上到下串行执行，前序步骤未完成，无法进入下一渲染阶段

---

## 二、分步拆解：每阶段流程、规则、底层逻辑

### 1\. HTML解析：构建DOM树

#### 1\.1 细化解析流程

网络字节流 → 编码转字符 → 词法拆分Token标签 → 语法组装节点 → 生成DOM树

1. **字节**：服务器返回的二进制网络数据包

2. **字符**：根据HTML charset编码，二进制转文本字符

3. **Token**：浏览器分词器拆分标签、属性、文本，生成标签令牌

4. **节点**：Token校验合规后，转化为Element/Text注释节点

5. **DOM树**：树形结构化节点，可供JS读写操作的文档对象模型

#### 1\.2 核心特性

- 默认自上而下逐行解析HTML

- 遇到普通\<script\>标签：**暂停HTML解析，优先执行JS**

### 2\. CSS解析：构建CSSOM树

#### 2\.1 执行流程

解析CSS样式文件/内联样式 → 过滤无效样式、兼容前缀处理 → 生成CSSOM样式规则树

#### 2\.2 核心硬性规则（高频考点）

**CSS解析阻塞页面渲染**：CSS未解析完成，无法生成Render树，页面空白无样式

解除阻塞方案：给link标签添加 `media` 媒体属性，匹配当前设备不生效则不阻塞渲染

#### 2\.3 补充特性

- CSS不会阻塞HTML解析，仅阻塞渲染

- CSSOM树包含所有元素默认样式\+自定义样式

### 3\. Render Tree渲染树（渲染核心载体）

#### 3\.1 生成规则

**Render Tree = 可见DOM节点 \+ 对应生效CSSOM样式**

#### 3\.2 关键筛选规则（易错点）

- 参与渲染：`visibility:hidden` 占位隐藏元素（占空间，参与渲染树）

- 不参与渲染：`display:none` 彻底隐藏元素、注释节点、meta无视觉节点

### 4\. Layout 布局（回流/重排）

#### 4\.1 定义

基于Render树，计算每一个可见元素**几何信息**：元素宽高、页面坐标、margin、定位位置

#### 4\.2 触发回流场景

- 修改元素宽高、margin、top/left布局属性

- 新增/删除DOM节点、修改页面视口大小

- 读取offsetWidth、scrollTop等布局属性

### 5\. Paint 绘制（重绘）

#### 5\.1 定义

根据Layout布局结果，填充像素信息：颜色、背景、阴影、边框，把元素绘制到对应图层

#### 5\.2 回流\&重绘区别

- 回流：改变位置大小，**一定触发重绘**，性能开销大

- 重绘：仅改颜色背景，不改变布局，**不会触发回流**，开销更小

### 6\. Composite 图层合成（最优性能步骤）

#### 6\.1 执行逻辑

浏览器分层管理页面元素，交由**GPU独立并行合成图层**，合并图层输出最终屏幕画面

#### 6\.2 性能最优操作

修改 `transform / opacity` 属性：仅触发合成，不回流、不重绘，前端动画最优方案

## 三、CSS \+ JS 双向阻塞关系（重难点\+代码案例）

### 3\.1 CSS 阻塞规则

✅ 规则1：CSS阻塞页面渲染，未加载解析完CSS，页面不会展示样式内容

✅ 规则2：CSS不阻塞HTML解析

✅ 解除阻塞：link标签配置media，非匹配媒体环境不阻塞渲染

#### 案例1：media属性解除CSS渲染阻塞

```html
<!-- 桌面端生效，移动端不加载、不阻塞渲染 -->
<link rel="stylesheet" href="pc.css" media="(min-width:768px)">
<!-- 打印样式，页面浏览时不阻塞渲染 -->
<link rel="stylesheet" href="print.css" media="print">
```

### 3\.2 JS 阻塞规则

✅ 规则1：原生同步\<script\> **阻塞HTML解析**，暂停DOM构建，执行完JS才继续解析HTML

✅ 规则2：JS会等待CSSOM构建完毕再执行（JS可读写样式，依赖CSSOM）

✅ 优化方案：async / defer 解除HTML解析阻塞

#### 案例2：三类script脚本阻塞对比实操

```html
<!-- 1. 同步脚本：阻塞HTML解析，顺序执行，阻塞渲染 -->
<script src="index.js"></script>

<!-- 2. defer延迟脚本：不阻塞HTML解析，DOM解析完成后按顺序执行 -->
<script src="a.js" defer></script>
<script src="b.js" defer></script>

<!-- 3. async异步脚本：不阻塞HTML解析，下载完成立即执行，执行顺序无序 -->
<script src="c.js" async></script>
```

#### 脚本核心对比表

|脚本类型|是否阻塞HTML解析|执行顺序|执行时机|
|---|---|---|---|
|原生同步script|✅ 阻塞|从上到下有序|下载完毕立刻执行|
|defer|❌ 不阻塞|从上到下有序|DOM解析完成后执行|
|async|❌ 不阻塞|下载优先，无序|资源下载完毕立刻执行|

## 四、高频易混概念总结

### 4\.1 三大树形结构区别

1. **DOM树**：仅存储HTML结构节点，无关样式

2. **CSSOM树**：仅存储全局样式规则，无关结构

3. **Render树**：结构\+样式结合，仅保留可视元素

### 4\.2 display:none 与 visibility:hidden 渲染区别

- display:none：无Render树节点，无布局、无绘制，完全不渲染

- visibility:hidden：存在Render树节点，占用布局空间，完成布局绘制，仅像素透明

### 4\.3 渲染三级开销排序（性能优先级）

合成Composite（最优） \< 重绘Paint \< 回流Layout（最差）

## 五、前端性能优化落地结论（背诵考点）

1. 外联CSS放在head内，避免页面后期回流闪动

2. 非首屏CSS添加media属性，解除渲染阻塞

3. 业务JS优先使用defer，第三方埋点SDK使用async

4. 动画优先使用transform/opacity，规避回流重绘

5. 避免频繁读写DOM布局属性，合并DOM修改操作
