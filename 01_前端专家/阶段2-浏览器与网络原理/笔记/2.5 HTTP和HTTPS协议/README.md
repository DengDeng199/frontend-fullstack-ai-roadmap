# 一、HTTP三大版本迭代对比（核心特性\+痛点）

## 1\. HTTP/1\.1（目前存量最广，浏览器默认兼容版本）

### ✅ 核心特性

1. **持久连接（长连接）**：默认开启 `Connection: keep-alive`，一次TCP连接可传输多次HTTP请求，不用每次请求都断开重连，减少TCP握手开销。

2. **管线化 Pipeline**：客户端可一次性批量发送多个请求，无需等待上一个响应返回，再发下一个请求。

### ❌ 致命痛点：队头阻塞（HOL阻塞）

**原理**：HTTP/1\.1 文本格式传输，同一TCP连接内，**请求有序发送、响应必须有序返回**。

**业务案例**：页面并行请求【图片A、图片B、图片C】，图片A服务器卡顿延迟响应，图片B、C即便资源就绪，也必须排队等待A响应完毕，才能返回数据，造成页面资源加载串行卡顿。

### 补充兜底方案

浏览器默认开启**域名并行限制**：同一个域名最多建立6\-8个TCP连接，缓解阻塞，但会占用端口、增加握手开销。

## 2\. HTTP/2（主流业务优选，CDN全站普及）

基于TCP协议改造，完全兼容HTTP语义，仅传输层重构，彻底解决应用层队头阻塞

### 四大核心特性（面试必背）

1. **二进制分帧**

    - 摒弃HTTP/1\.1文本明文传输，拆分为：帧\(Frame\)、消息、流三层结构

    - 二进制解析效率远高于文本解析，容错性更强

2. **多路复用**

    - 单个TCP连接，可并行收发**不限量**请求响应，互不干扰

    - 彻底解决HTTP应用层队头阻塞，请求带专属流ID，响应可乱序返回

3. **HPACK头部压缩**

    - HTTP请求头部cookie、请求头重复度极高，HTTP/1\.1每次请求全量携带

    - HPACK建立头部索引表，重复头部仅传索引，大幅减少请求体积，弱网提速明显

4. **服务器推送 Server Push**

    - 客户端未发起请求，服务端主动推送关联静态资源（html加载后主动推送css、js）

    - 优化首屏资源加载速度，避免客户端解析html后再二次请求资源

### 遗留问题

底层依旧基于TCP，**TCP传输层队头阻塞无法解决**：TCP丢包时，整条连接所有数据流都会等待重传，全部阻塞。

## 3\. HTTP/3（QUIC协议，新一代标准）

### 核心底层改动

彻底抛弃TCP，**基于UDP协议封装QUIC协议**，上层保留HTTP语义，向下重构传输层。

### 核心优势

1. 彻底解决**TCP传输层队头阻塞**：UDP面向报文独立传输，某一条流丢包重传，不会影响其他数据流

2. 0\-RTT/1\-RTT握手：相比TCP\+TLS四次交互，大幅减少握手时延，移动端弱网优化极强

3. 自带加密、连接迁移：切换wifi/4G网络，连接不中断，适配移动端网络波动场景

### 业务现状

大厂全站部署（抖音、B站、谷歌），中小型项目逐步落地，浏览器全兼容，服务端配置成本偏高。

## 版本极简对比表

|协议版本|底层协议|队头阻塞层级|核心优化|
|---|---|---|---|
|HTTP/1\.1|TCP|应用层阻塞|长连接、管线化|
|HTTP/2|TCP|传输层阻塞|多路复用、头部压缩|
|HTTP/3|UDP\(QUIC\)|无队头阻塞|0握手、独立报文传输|

# 二、HTTPS 完整握手流程（面试必考四步）

HTTPS = HTTP 应用协议 \+ TCP传输协议 \+ TLS加密协议，整体分为两大阶段：TCP三次握手 \+ TLS加密握手

## 1\. 阶段一：TCP三次握手（建立可靠传输通道）

目的：建立双向可靠连接，确认客户端、服务端收发数据能力正常

1. 客户端 → 服务端：SYN报文（请求建立连接）

2. 服务端 → 客户端：SYN\+ACK报文（同意连接，回执确认）

3. 客户端 → 服务端：ACK报文（连接建立完成）

💡 区别：HTTP明文只需要TCP握手，HTTPS需要额外TLS加密握手

## 2\. 阶段二：TLS1\.2 标准四步握手（主流版本）

1. **Client Hello**：客户端发送支持的TLS版本、加密套件、随机数、压缩算法

2. **Server Hello**：服务端选定加密套件、TLS版本、下发服务端随机数、下发CA数字证书

3. **证书校验**：客户端校验证书域名、有效期、颁发机构、签名合法性，防止证书伪造

4. **密钥交换\+协商会话密钥**：客户端生成随机预密钥，用服务端公钥加密发送，服务端私钥解密，双方协商出一致**会话密钥**

握手结束：双方使用会话密钥，进行对称加密传输业务数据

## 3\. 两类加密方式辨析（高频面试）

### 非对称加密（握手阶段使用）

- 密钥成对：公钥\(公开加密\)、私钥\(私有解密\)

- 特点：加密安全、运算速度极慢

- 用途：仅用来传输「对称加密会话密钥」，不传输业务数据

### 对称加密（业务数据传输全程使用）

- 特点：客户端服务端共用同一个会话密钥，加解密速度极快

- 用途：网页接口、图片、文本所有业务数据加密传输

HTTPS加密核心逻辑：非对称加密保密钥安全，对称加密保传输速度，二者搭配使用

# 三、HTTP 缓存体系（强缓存 \+ 协商缓存）

缓存目的：减少HTTP请求、降低服务器压力、提升页面加载速度，优先级：**强缓存优先，失效再走协商缓存**

## 1\. 强缓存（不发请求，直接读取本地缓存，状态码200 from memory/disk cache）

无需和服务端交互，浏览器直接复用本地资源，性能最优

### 两组控制字段

#### ① Expires（HTTP/1\.0 旧版，已淘汰）

- 值：服务端返回的**绝对过期时间**

- 坑点：依赖客户端本地时间，修改电脑时间，缓存直接失效，兼容性差

#### ② Cache\-Control（HTTP/1\.1 新版，优先使用）

相对时间，不受本地时间影响，常用指令业务案例

```http
# 业务常用响应头配置
# 1. 资源缓存30天，公共缓存（CDN、浏览器均可缓存）
Cache-Control: public, max-age=2592000
# 2. 仅浏览器缓存，CDN不缓存（用户私有资源）
Cache-Control: private, max-age=86400
# 3. 禁止强缓存，每次必发请求
Cache-Control: no-cache
# 4. 彻底禁用所有缓存，直接获取新资源
Cache-Control: no-store
```

## 2\. 协商缓存（发送请求，服务端校验，命中返回304 Not Modified）

强缓存过期后触发，携带资源标识询问服务端：资源是否修改，未修改则复用本地缓存

### 两组成对字段

#### ① Last\-Modified / If\-Modified\-Since（HTTP/1\.0）

- 服务端响应头：Last\-Modified=资源最后修改时间

- 客户端请求头：If\-Modified\-Since 带回上次修改时间比对

- 坑点：只能精确到秒，1秒内多次修改资源，无法识别更新

#### ② ETag / If\-None\-Match（HTTP/1\.1 优选）

- ETag：服务端生成资源唯一指纹哈希值，文件改动哈希必变

- 精度极高，秒级内修改、文件名不变内容修改，均可识别

- 优先级高于 Last\-Modified

## 3\. 缓存完整执行流程

1. 请求资源 → 先判断Cache\-Control是否在有效期 ✅命中：读取本地缓存200

2. ❌强缓存失效 → 携带ETag/Last\-Modified发请求协商

3. 服务端比对：资源未修改 → 返回304，复用本地缓存

4. 服务端比对：资源已修改 → 返回200，下发新资源\+新缓存标识

## 4\. 业务实操缓存配置案例

- 静态图片、图标：Cache\-Control: max\-age=31536000（缓存一年，永久复用）

- 迭代js/css：打包添加哈希后缀，配合长缓存，文件变更哈希变更自动更新

- 首页html：Cache\-Control: no\-cache，每次协商缓存，保证页面及时更新

# 四、全文面试速记易错点

1. HTTP/1\.1队头阻塞：应用层；HTTP/2队头阻塞：TCP传输层；HTTP/3无阻塞

2. 多路复用：HTTP/2独有，单TCP连接多请求并行

3. HTTPS：TCP三次握手 \+ TLS加密握手，非对称加密传密钥，对称加密传数据

4. 缓存优先级：Cache\-Control \> Expires；ETag \> Last\-Modified；强缓存 \> 协商缓存

5. 状态码：200\(本地强缓存\)、304\(协商缓存命中\)
