# 一、TCP 核心特性前置认知

TCP（传输控制协议）是**面向连接、可靠、有序、字节流**的传输层协议，核心保障：数据不丢、不乱、不重复、完整到达。

核心前置概念：

- **SYN**：连接请求报文（发起握手）

- **ACK**：确认应答报文（收到数据/请求）

- **FIN**：断开连接报文（发起挥手）

- **序列号**：保证数据有序、去重

- **确认号**：告知对方「我已收到哪些数据」

# 二、TCP 三次握手（建立可靠连接）

## 1\. 完整流程（标准顺序）

核心目的：**双向确认收发能力正常**（客户端能发能收、服务端能发能收），建立可靠传输通道。

1. **第一次握手：客户端 → 服务端（SYN）**

    - 客户端随机生成初始序列号，发送 SYN 报文，请求建立连接

    - 状态：客户端 **SYN\_SENT**（同步已发送）

2. **第二次握手：服务端 → 客户端（SYN\+ACK）**

    - 服务端收到 SYN，确认客户端发送能力正常

    - 服务端生成自身初始序列号，返回 SYN\+ACK 报文

    - 状态：服务端**SYN\_RCVD**（同步已接收）

3. **第三次握手：客户端 → 服务端（ACK）**

    - 客户端收到 SYN\+ACK，确认服务端收发能力正常

    - 发送 ACK 确认报文，连接正式建立

    - 双方状态切换为 **ESTABLISHED**（连接已建立，可传输数据）

## 2\. 通俗生活化案例

场景：两人打电话确认通话正常

1. 你（客户端）：喂，能听到吗？（SYN）

2. 对方（服务端）：我能听到，你能听到我吗？（SYN\+ACK）

3. 你（客户端）：我可以听到！（ACK）

三次确认完成，双方正式开始对话（传输数据）。

## 3\. 面试高频问题：为什么是三次，不是两次？

**核心原因：防止失效连接请求占用服务端资源**

若只有两次握手：客户端失效、延迟的 SYN 请求到达服务端，服务端直接建立连接、占用端口资源，无人通信导致资源浪费。

三次握手可通过最后一次 ACK 校验，过滤无效连接请求。

# 三、TCP 四次挥手（断开连接）

## 1\. 核心原理

TCP 连接是**全双工**通道，读写通道相互独立，因此断开需要**双向分别关闭**，无法一次性断开，固定四次交互。

## 2\. 完整流程（标准顺序）

1. **第一次挥手：主动方 → 被动方（FIN）**

    - 主动关闭方（客户端/服务端均可）发送 FIN 报文，表示「我没有数据要发了，请求关闭发送通道」

    - 状态：主动方 **FIN\_WAIT\_1**

2. **第二次挥手：被动方 → 主动方（ACK）**

    - 被动方收到 FIN，返回 ACK 确认

    - 此时状态：**半关闭状态**（主动方不能发数据，但可以收数据）

    - 主动方 **FIN\_WAIT\_2**，被动方 **CLOSE\_WAIT**

3. **第三次挥手：被动方 → 主动方（FIN）**

    - 被动方数据全部传输完毕，发送 FIN 报文，关闭自身发送通道

    - 状态：被动方 **LAST\_ACK**

4. **第四次挥手：主动方 → 被动方（ACK）**

    - 主动方返回 ACK 确认，等待超时释放资源

    - 双方连接彻底关闭，状态变为 **CLOSED**

## 3\. 通俗案例

场景：两人结束通话

1. 你：我说完了，我不说话了（FIN）

2. 对方：收到，我知道你不说话了（ACK）—— 对方还可以继续说话

3. 对方：我也说完了，我也不说话了（FIN）

4. 你：收到，通话结束（ACK）

## 4\. 面试考点：为什么挥手需要四次？

因为 TCP 读写通道独立，**关闭发送通道和接收通道需要分开确认**。第二次挥手仅确认关闭对方的发送权限，本方可能还有残留数据需要传输，因此必须多两次交互。

# 四、TCP 拥塞控制（四大核心算法）

作用：防止网络拥堵、缓冲区溢出、大量丢包，**动态调整发送窗口大小**，适配网络带宽，是 TCP 流畅传输的核心。

## 1\. 慢启动 Slow Start

- **核心逻辑**：连接刚建立，不清楚网络带宽，**先发少量数据试探**

- **增长规则**：拥塞窗口 cwnd **指数增长**（1→2→4→8）

- **终止条件**：达到「慢启动阈值 ssthresh」，退出慢启动，进入拥塞避免

**案例**：刚上车的高速路，先低速试探路况，确认通畅再提速。

## 2\. 拥塞避免 Congestion Avoidance

- **核心逻辑**：网络通畅后，不再激进提速，**稳步线性增长**

- **增长规则**：每个往返时间 RTT，cwnd 只 \+1

- **目的**：避免瞬间流量暴涨，冲垮网络链路

**案例**：路况通畅，保持匀速行驶，不超速、不拥堵。

## 3\. 快重传 Fast Retransmit

- **场景**：个别报文丢包，不等待超时重传

- **规则**：连续收到 **3 次重复 ACK**，立刻判定报文丢失，立即重传缺失报文

- **优势**：大幅减少超时等待时间，提升弱网传输效率

**案例**：连续三次听不到对方回应，立刻重发消息，不用干等超时。

## 4\. 快恢复 Fast Recovery

- **触发时机**：快重传执行后，不退回慢启动

- **核心逻辑**：将 ssthresh、cwnd 减半，直接进入拥塞避免阶段

- **目的**：网络轻微拥堵，不用从零试探，快速恢复传输速度

**案例**：轻微堵车不用停车重启，减速匀速通行即可。

## 5\. 四算法流转闭环

新连接 → 慢启动（指数提速）→ 达阈值 → 拥塞避免（匀速提速）→ 丢包触发快重传 → 快恢复降速维稳 → 稳定传输

# 五、IP 协议基础（网络层核心）

## 1\. 核心作用

IP 协议是 TCP/IP 体系**网络层核心协议**，负责：**寻址 \+ 路由转发**，给每一台网络设备分配唯一地址，规划数据传输路径。

## 2\. 核心特性

- **无连接、不可靠**：只负责转发，不保证送达、不重传、不排序（可靠传输由上层 TCP 实现）

- 提供唯一 IP 标识：区分全网设备

- 支持路由分片：超大数据包自动分片传输，到达后重组

## 3\. IP 地址分类（基础认知）

- **IPv4**：32位，四段十进制（如 192\.168\.1\.1），地址资源枯竭

- **IPv6**：128位，八段十六进制，海量地址，解决枯竭问题，安全性更高

## 4\. 通俗分层案例

IP 地址 = 小区楼栋门牌号（定位设备位置）

TCP 端口 = 房间号（定位设备内的具体服务）

二者结合，才能精准把数据送到指定设备的指定服务。

# 六、DNS 解析完整流程（域名 → IP）

核心作用：人类记忆域名（www\.baidu\.com），机器识别 IP，DNS 负责**域名与 IP 映射解析**，是网页访问的第一步。

## 1\. 完整递归\+迭代解析流程（标准 6 步）

1. **浏览器缓存查询**：先查浏览器本地缓存，有记录直接返回 IP

2. **系统缓存查询**：浏览器无缓存，查询操作系统 hosts 文件、本地 DNS 缓存

3. **本地 DNS 服务器**：无缓存，向运营商本地 DNS 服务器发起迭代查询

4. **根域名服务器**：本地 DNS 访问根服务器，根服务器返回顶级域名服务器地址

5. **顶级域名服务器**（\.com/\.cn/\.net）：返回目标权威域名服务器地址

6. **权威域名服务器**：查询最终解析记录，返回目标服务器真实 IP，浏览器缓存后发起 TCP 请求

## 2\. 核心概念区分

- **递归查询**：浏览器 → 本地DNS（我要最终结果，你帮我查完）

- **迭代查询**：本地DNS → 根/顶级/权威服务器（查不到就告诉我下一级地址，我自己查）

## 3\. 业务场景案例

首次访问 `www.baidu.com`：走完完整6级解析流程

二次访问：读取浏览器缓存，**毫秒级直达**，无需重复解析

## 4\. 常见优化方案

- DNS 预解析：`<link rel="dns-prefetch" href="https://xxx.com">`

- 配置合理 DNS 缓存时间，减少重复解析开销

- 使用公共 DNS（114\.114\.114\.114、8\.8\.8\.8）提升解析稳定性
