## 1. 可视化设计的底层目标

> 可视化的唯一目的是**「让人在最短时间、最准确地理解数据要表达的结论」**。一切设计决策都要回到这句话。

```
✅ 好的可视化：结论一眼可见、对比清晰、无歧义
❌ 坏的可视化：华而不实、误导读者、信息过载、配色刺眼
```

经典反例警示（你应该避免的）：

| 反模式 | 后果 | 修正 |
|--------|------|------|
| 截断 Y 轴起点 | 夸大差异，误导结论 | 从 0 开始（柱状图） |
| 3D 饼图 | 角度畸变难比较 | 用条形图或平面饼图 |
| 同时用 6 种高饱和色 | 视觉混乱 | 统一色板 + 重点色 |
| 双 Y 轴强关联 | 虚假相关 | 拆分或用归一化 |
| 过多装饰（阴影/网格/图标） | 干扰数据 | 精简到只剩数据 |

> **核心心法**：可视化是「数据翻译」，不是「美术创作」。炫技的 3D 动效若削弱了数据可读性，就是失败的。

## 2. 色彩理论

### 2.1 HSL 三要素

```
H 色相 (Hue)      → 红/绿/蓝等「是什么颜色」
S 饱和度 (Sat)    → 颜色的「纯度/鲜艳度」
L 明度 (Light)    → 颜色的「明暗程度」
```

> 为什么用 HSL 而不是 RGB？**HSL 是人类直觉的**——「调亮一点」改 L，「换个色系」改 H，比改 RGB 三原色直观得多。D3 的 `d3.hsl()`、ECharts 的 `color` 都基于此思考。

### 2.2 三种数据配色类型（最重要的选型）

| 类型 | 适用数据 | 配色方式 | 例子 |
|------|---------|---------|------|
| **分类色（Categorical）** | 无顺序的类别 | 对比强、可区分的不同色相 | 不同设备类型、不同站点 |
| **顺序色（Sequential）** | 有大小顺序的数值 | 同一色相由浅到深 | 温度、信号强度 |
| **发散色（Diverging）** | 有中心基准的正负偏离 | 两端不同色、中间浅色 | 盈亏、偏差（±） |

```
分类：  🔴 🟢 🔵 🟡 🟣     （色相互异，无大小含义）
顺序：  ⬜ ⬜ ⬜ ⬛ ⬛         （同色系由浅到深，越大越深）
发散：  🔵 🟦 ⬜ 🟥 🔴       （中间浅，两边分冷暖）
```

### 2.3 色盲友好设计

> 全球约 **8% 男性、0.5% 女性** 有某种色觉缺陷（最常见为红绿色盲）。只用「红绿」区分数据会让他们完全看不出差异。

| 做法 | 说明 |
|------|------|
| 避免红绿配对 | 红绿色盲看红绿都发灰 |
| 用 **色相 + 形状/纹理** 双重编码 | 颜色之外加图标/线型差异 |
| 用蓝橙对比 | 蓝橙是色盲最易区分的组合 |
| 调高明度对比 | 不依赖色相也能区分深浅 |
| 用在线工具校验 | Coblis、Color Oracle 模拟色盲效果 |

```js
// 色盲友好的顺序色板（蓝→橙，避免红绿）
const safePalette = ['#4477aa', '#66ccee', '#ffffff', '#ee6677', '#aa3377'];
```

### 2.4 大屏暗色主题配色要点

```
背景：近黑深蓝（#0a0e1a / #0e1e3a），营造科技感、降低眩光
数据色：高饱和亮色（#4f9dff / #00d4c8 / #ffb454），暗底才醒目
网格线：低对比（rgba(255,255,255,0.06)），绝不抢数据
文字：中灰蓝（#c9d3e0），非纯白（纯白刺眼）
重点色：告警用橙红 #ff5d5d，突出但不泛滥
```

## 3. 信息密度与数据墨水比

### 3.1 数据墨水比（Data-Ink Ratio）

> 由 Edward Tufte 提出：**图表中「承载数据的墨水」÷「总墨水」**。比例越高，信息效率越高。

```
数据墨水 = 真正表示数值的线条/面积（柱体、折线、点）
非数据墨水 = 网格线、边框、阴影、背景、装饰图标
```

| 提高数据墨水比 | 降低（删掉） |
|---------------|-------------|
| 直接画柱状/折线 | 多余的外边框 |
| 用数据点的密集度表密度 | 3D 透视阴影 |
| 网格线只留必要刻度 | 花哨的背景图 |
| 标签直接标在图形旁 | 图例冗余（能直接标注就别用图例） |

> **原则**：每删掉一处非数据墨水，图表反而更清晰。但「网格线/坐标轴」属于**必要参考系**，不能全删——要在「可读性」与「简洁」间平衡。

### 3.2 信息密度平衡

- **低密度（大屏概览）**：每屏 1~3 个核心结论，字大图简，远看即懂。
- **高密度（分析钻取）**：表格/明细/多维，供专家细看。
- 同一 dashboard 用 **概览 → 下钻** 分层承载不同密度，别一屏塞满。

## 4. 交互设计

> 静态图只能回答一个问题；交互让一张图回答多个问题。但要「克制」——交互多了反而增加认知负担。

### 4.1 四类基础交互

| 交互 | 作用 | 实现 |
|------|------|------|
| **Tooltip** | 悬浮看精确值 | ECharts `tooltip` / D3 `mouseover` |
| **高亮联动** | 跨图联动聚焦 | ECharts `connect` / D3 事件广播 |
| **缩放平移** | 看局部/全局 | ECharts `dataZoom`/`roam` / D3 `zoom` |
| **下钻（drill-down）** | 从汇总到明细 | 监听 click → 替换数据/地图 |

### 4.2 联动示例（ECharts 多图联动）

```js
// 两个图表共享 tooltip 与高亮
const chartA = echarts.init(domA);
const chartB = echarts.init(domB);
echarts.connect([chartA, chartB]); // 联动：一处 hover，多处响应

// 或手动广播：A 的 hover 事件同步给 B
chartA.on('updateAxisPointer', (e) => {
  chartB.dispatchAction({ type: 'showTip', seriesIndex: 0, dataIndex: e.dataIndex });
});
```

### 4.3 交互设计原则

1. **默认展示结论**：首屏就给出核心洞察，交互是「锦上添花」不是「必须操作才看得到」。
2. **即时反馈**：hover 0 延迟，避免用户猜「点哪」。
3. **可撤销/返回**：下钻后要有「返回上一层」。
4. **不要过度动画**：转场 > 500ms 反而拖慢分析节奏。
5. **移动端照顾手势**：缩放手势、点按区域足够大。

## 5. 图表选型决策树

> 选错图表类型，数据再对也会被误读。**先想清「我想让用户比较什么」再选图**。

```
你想表达什么？
│
├─ 比较数值大小（类别间）──▶ 柱状图 / 条形图
│                              （类别<10 用柱状，>10 或名字长用条形）
│
├─ 看随时间变化趋势 ───────▶ 折线图（多序列）/ 面积图（含总量感）
│
├─ 看占比 / 构成 ──────────▶ 饼图（≤5 类）/ 环形图（中心可放总量）
│                              堆叠柱状图（可看总量+构成）
│
├─ 看分布 / 相关性 ────────▶ 散点图 / 气泡图（加第三维用大小）
│                              直方图（单变量分布）/ 箱线图（分布+离群）
│
├─ 看关系 / 网络 ──────────▶ 力导向图（D3）/ 桑基图（流向）
│
├─ 看地理空间 ─────────────▶ 地图 / 热力图 / 飞线
│
└─ 看层级 / 构成树 ────────▶ 树图 / 旭日图 / 矩形树图
```

### 5.1 常见误用与修正

| 误用 | 问题 | 改用 |
|------|------|------|
| 用饼图展示 12 个类别 | 难比较、挤成一团 | 条形图（按值排序） |
| 用 3D 柱状图 | 透视畸变、难读值 | 2D 柱状/条形 |
| 用折线图比类别大小 | 折线暗示「连续性」 | 柱状图 |
| 用面积图比单点值 | 面积=值²，易误读 | 柱状图 |
| Y 轴不从头开始 | 夸大差异 | 从 0 开始（柱状） |

### 5.2 「按数量级」速查

| 数据特征 | 首选 |
|---------|------|
| 类别 < 5，看占比 | 饼图 |
| 类别 5~10，看比较 | 柱状图 |
| 类别 > 10，或名字长 | 横向条形图 |
| 时间序列，看趋势 | 折线图 |
| 部分-整体，且要多层 | 堆叠面积 / 堆叠柱 |
| 两变量相关性 | 散点图 |
| 三变量 | 气泡图（x/y/大小） |

## 6. 无障碍访问

> 可视化不是只给视力正常的人看的。无障碍（a11y）让色盲、键盘用户、读屏软件用户也能获取信息。

### 6.1 三个抓手

| 维度 | 做法 |
|------|------|
| **ARIA 标签** | 给图表容器加 `role="img"` + `aria-label` 描述结论；用 `aria-hidden` 隐藏纯装饰 SVG |
| **键盘导航** | 图例/数据点可被 Tab 聚焦，Enter 触发选中；不依赖纯鼠标 hover |
| **高对比度** | 文字与背景对比度 ≥ 4.5:1（WCAG AA）；不只靠颜色编码 |

### 6.2 实战清单（HTML/SVG 层面）

```html
<!-- 图表容器：读屏用户能「听到」图表结论 -->
<div role="img"
     aria-label="2026年各监测站在线率：成都 98%、北京 95%、广州 91%">
  <svg>...</svg>
</div>

<!-- 颜色之外加文字/形状编码（色盲可读） -->
<rect fill="#4f9dff" aria-label="在线"></rect>
<rect fill="#ff5d5d" aria-label="离线"></rect>
```

```js
// ECharts 无障碍：开启后会生成隐藏的 DOM 描述
chart.setOption({
  aria: {
    show: true,                     // 开启无障碍
    description: '本图表展示各监测站信号强度趋势',
  },
});
```

### 6.3 为什么重要

- **合规**：政府/金融类项目常要求 WCAG 标准。
- **可用性**：强光环境（大屏现场）下高对比度更易读。
- **不依赖单一编码**：颜色 + 形状 + 文字 = 人人可读。

## 7. caidiaweb 大屏设计实践

### 7.1 大屏设计「三不原则」

```
① 不堆砌：一屏 3~6 个模块，每个模块 1 个核心结论
② 不刺眼：暗底 + 亮数据色 + 低对比网格，长时间观看不疲劳
③ 不误导：Y 轴从 0、不用 3D 畸变、双 Y 轴慎用
```

### 7.2 配色方案落地

```js
// caidiaweb 统一大屏色板（已注册为 ECharts 主题）
export const palette = {
  bg: '#0a0e1a',
  panel: '#0e1e3a',
  text: '#c9d3e0',
  series: ['#4f9dff', '#00d4c8', '#ffb454', '#ff5d5d', '#a78bfa'],
  warn: '#ff5d5d',
  grid: 'rgba(255,255,255,0.06)',
};

// 分类数据用 series；强度用顺序（蓝→青）；偏差用发散（蓝-白-红）
```

### 7.3 图表选型映射

| caidiaweb 模块 | 数据意图 | 选型 |
|---------------|---------|------|
| 各站信号强度对比 | 比较 | 条形图（站名长，横向） |
| 24h 信号趋势 | 趋势 | 折线图 |
| 设备状态构成 | 占比 | 环形图（中心放总数） |
| 频段占用分布 | 分布 | 热力图 |
| 干扰源关联 | 关系 | 力导向图 |
| 监测站地理 | 地理 | 地图 + 涟漪点 |

### 7.4 落地检查清单

- [ ] 配色用蓝橙/顺序/发散三类，避免红绿配对
- [ ] 色板统一（ECharts theme / CSS 变量），不随手写死颜色
- [ ] Y 轴从 0 起（柱状）；折线可截断但要标注
- [ ] 网格线低对比，不抢数据
- [ ] Tooltip/高亮联动让数据可读，不靠纯装饰
- [ ] 大屏开启 ECharts `aria`，关键结论有文字兜底
- [ ] 首屏直接呈现核心结论，下钻才看明细

## 8. 面试考点

### Q1：数据墨水比是什么？怎么提高？
Tufte 提出的指标：图表中「承载数据的墨水」÷「总墨水」。提高方法——删掉多余边框/阴影/装饰，网格线只留必要刻度，标签直接标在图形旁而非用图例。但坐标轴/网格属于必要参考系不能全删。

### Q2：分类色、顺序色、发散色怎么区分使用？
分类色用于无顺序的类别（不同设备），用对比色相；顺序色用于有大小顺序的数值（温度），用同色系由浅到深；发散色用于有中心基准的正负偏离（盈亏），用两端冷暖、中间浅。

### Q3：为什么色盲友好设计重要？怎么做？
约 8% 男性有红绿色盲，只用红绿区分数据他们看不出。做法：避免红绿配对、用蓝橙对比、色相+形状/纹理双重编码、提高明度对比、用 Coblis 等工具校验。

### Q4：怎么选图表类型？给个决策思路。
先明确「想让用户比较什么」：比较大小→柱状/条形；趋势→折线；占比→饼/环/堆叠柱；分布→散点/直方图；关系→力导向/桑基；地理→地图；层级→树/旭日。再按类别数量微调（>10 用条形）。

### Q5：有哪些常见图表误用？
饼图展示 >5 类、3D 柱状图透视畸变、折线图比类别大小、面积图比单点值、Y 轴不从头开始夸大差异。核心是「图表类型要匹配数据语义」。

### Q6：可视化的无障碍（a11y）包含什么？
ARIA 标签（role="img" + aria-label 描述结论）、键盘导航（图例/点可 Tab 聚焦）、高对比度（≥4.5:1，不只用颜色编码）。ECharts 可开 `aria.show` 生成隐藏描述。

### Q7：大屏暗色主题配色原则？
近黑深蓝背景降低眩光；数据用高饱和亮色（暗底才醒目）；网格线低对比不抢数据；文字中灰蓝非纯白；告警用橙红但不泛滥。整体统一色板，避免随手写死颜色。

### Q8：交互设计要遵循什么原则？
默认展示结论（交互是锦上添花）、即时反馈（hover 0 延迟）、可撤销返回（下钻有返回）、不过度动画（转场 <500ms）、移动端照顾手势。交互多了反而增加认知负担，要克制。

---

> **本章小结**：可视化设计 = 色彩（三类配色 + 色盲友好）+ 信息密度（数据墨水比）+ 交互（Tooltip/联动/缩放/下钻）+ 选型（先想比什么再选图）+ 无障碍（ARIA/键盘/高对比）。记住底层目标——**让人在最短时间最准确地读懂数据结论**。下一章 **5.7 大屏可视化** 将把前 6 章所有技术（SVG/Canvas/WebGL/ECharts/D3/设计原则）落到「大屏」这一综合载体上。
