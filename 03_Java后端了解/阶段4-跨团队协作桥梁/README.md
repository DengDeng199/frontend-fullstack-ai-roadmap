## 学习内容

### 4.1 接口文档规范

- **Swagger/OpenAPI 规范**：
  - 能看懂 Swagger UI 中的接口定义
  - 能理解请求参数、响应格式、状态码
  - 能验证接口文档与实际实现的一致性
- **接口需求文档写作**：
  - 请求方法 / URL / 参数 / 响应格式
  - 错误码定义
  - 分页规范
  - 版本策略

### 4.2 鉴权流程理解

- **JWT 完整流程**：
  ```
  前端登录 → 后端验证 → 返回 JWT → 前端存储
  → 每次请求 Header 携带 → 后端验证 JWT → 返回数据
  → Token 过期 → Refresh Token 刷新
  ```
- **Spring Security 基础**：
  - 过滤器链（Filter Chain）
  - 权限模型（RBAC — 角色/权限）
  - 能理解为什么某个接口返回 401/403
- **Cookie vs Token vs Session**：
  - 各自的优缺点
  - 公司使用的方案及原因

### 4.3 日志定位能力

- **Java 日志格式**：
  ```
  2026-05-26 10:00:00.123 [http-nio-8080-exec-1] INFO  c.e.service.UserService - 查询用户: id=123
  ```
  - 时间戳 / 线程名 / 日志级别 / 类名 / 消息
- **异常堆栈阅读**：
  ```
  java.lang.NullPointerException
    at com.example.service.UserService.getUser(UserService.java:45)
    at com.example.controller.UserController.findById(UserController.java:23)
    ...
  Caused by: java.sql.SQLException: Connection refused
    at ...
  ```
  - 从上往下找自己代码的第一行
  - 注意 Caused by（根因）
- **常见后端错误类型**：
  - NullPointerException — 空指针
  - SQLException — 数据库错误
  - TimeoutException — 超时
  - JsonProcessingException — JSON 序列化错误

### 4.4 技术沟通能力

**用后端听得懂的语言描述前端需求**：
- 不要说"我需要数据格式是..." → 说"请返回如下 JSON 结构：{ code: 200, data: { list: [], total: 0 } }"
- 不要说"这个接口太慢了" → 说"该接口 P99 响应时间 > 2s，影响用户体验"
- 不要说"数据不对" → 说"预期字段 status 为数字 1/2，实际返回字符串 'active'"

**能评估需求的技术复杂度**：
- 前端需要展示什么 → 需要后端提供什么数据 → 是否需要新建接口 → 预计影响范围

**能提前发现接口设计问题**：
- 分页参数不统一（page/size vs pageNum/pageSize）
- 日期格式不统一（时间戳 vs ISO 8601 vs yyyy-MM-dd）
- 字段命名不统一（snake_case vs camelCase）
- 缺少必要的字段（如列表接口不返回 total）
- 错误码体系不规范

### 4.5 常见问题排查速查

| 问题 | 可能原因 | 排查方向 |
|------|---------|---------|
| 接口返回 400 | 参数不匹配 | 检查请求参数格式/类型 |
| 接口返回 401 | 未登录/Token 过期 | 检查 Authorization Header |
| 接口返回 403 | 无权限 | 检查角色/权限配置 |
| 接口返回 500 | 后端异常 | 看后端日志异常堆栈 |
| 数据字段缺失 | 后端没返回/字段名不一致 | 对比 Swagger 文档和实际返回 |
| 日期时区不对 | 前后端时区不一致 | 确认使用 UTC 还是本地时间 |
| 文件上传失败 | 大小超限/类型不支持 | 检查 Nginx/后端大小限制 |
| 中文乱码 | 编码不一致 | 确认 UTF-8 编码 |
